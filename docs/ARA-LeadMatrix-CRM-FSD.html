5Functional Overview

ARA Lead Matrix is a multi-tenant performance-marketing CRM that connects ad spend, lead generation, telecalling and results into one live system, exposed through two authenticated experiences.

Functionally, the platform performs five continuous jobs:

FunctionBehaviour
AcquireAutomatically ingests campaigns, ad sets, ads, daily insights, lead forms and leads from Meta and Google Ads on a recurring schedule; Meta leads additionally arrive in real time via signed webhooks.
Attribute & ReconcileNormalises each lead to a source/platform, keeps counts consistent via a single metrics service, and reconciles every ad-spend rupee into an immutable billing ledger.
WorkPresents leads on a validated 26-column telecalling sheet where users log calls, responses, appointments and follow-ups, with automation for reminders, history and dormancy.
ReportRenders live dashboards, EOD and Monthly Abstract reports, comparisons and analytics — auto-refreshing and identical across every surface.
GovernEnforces two role systems, strict tenant isolation, encrypted secrets, and full audit/observability of syncs, raw leads and financial entries.

5.1 Functional principles enforced by the system

  • Single source of truth — all KPIs are computed by one metrics service, so every screen and export agrees.
  • Idempotency — lead ingestion and billing writes are safe to repeat; duplicates are impossible.
  • Tenant isolation — a portal user's client is derived from their token, never the request.
  • Automation over manual effort — reminders, history and dormancy flags are computed server-side before every save.
  • Graceful degradation — when an integration (Meta billing, AI) is unavailable, the affected panel degrades without breaking the page.

6Application Overview

6.1 Architecture (functional view)

LayerResponsibility
Front-end SPAReact single-page app; two shells (agency console, client portal); Redux state; lazy-loaded routes; offline curtain; auto-refresh timers.
API layerExpress REST API under /api; JWT auth (header or cookie); per-route role/permission & tenant guards; centralised error handler; two-tier rate limiting.
Service layerMeta & Google ad services, single metrics service, billing reconcilers, telecaller automation, Excel engine, AI client.
Background jobsCron scheduler (Google + Meta sync, staggered), webhook receiver, retry worker, zombie-run cleanup.
Data storeMongoDB (Cosmos DB) — 24 collections; unique/compound indexes enforce idempotency and integrity.
External systemsMeta Graph API, Google Ads API, Google Gemini.

6.2 The two experiences

ExperienceEntry & tokenScope
Agency Console/login · token/userAll clients; capabilities gated by staff role + permissions.
Client Portal/client-login · clientTokenOne client only; capabilities gated by portal role (admin / telecaller) + strict tenant scope.

6.3 Environment & conventions

  • Currency: INR (₹). Business day: IST (Asia/Kolkata) for all daily roll-ups.
  • Response shape: JSON { success: boolean, data|error|message }.
  • Auth transport: Authorization: Bearer <jwt> or secure httpOnly token cookie (SameSite=None; Secure).
  • Session default: access token 30 days; refresh token 90 days (issued; used long-lived access token in practice).

9Navigation Flow

9.1 Agency console navigation

/login▶ Authenticated?▶ First permitted page (default Dashboard)▶ Sidebar: Dashboard · Leads Check · Clients · Total Leads · Ads Comparison · Analytics · Settings · [Client Portal · Access Mgmt]

The sidebar renders only pages the user's permissions allow. ProtectedRoute validates the session and page permission on every navigation; unauthorised access redirects to the first permitted page. Deep links (Dashboard client card, Clients row) route to /client-ads/:clientId.

9.2 Client portal navigation

/client-login▶ Role?▶ admin → Home · Leads · EOD · Google · Meta · Users / telecaller → Home · Leads · EOD

Telecaller-restricted tabs are both hidden and defended by redirect if reached directly. The portal shell carries a collapsible sidebar, client name and Logout.

9.3 Cross-cutting navigation behaviour

BehaviourDetail
Lazy loadingEvery page is code-split and prefetched after boot for instant subsequent navigation.
Offline curtainLoss of connectivity shows a full-screen OfflineScreen; restores automatically.
Deep-link filtersDate range, client and month selections are URL-encoded so views are shareable/bookmarkable.
Drill-throughDashboard tiles & source cards, and report figures, link to the Leads view pre-filtered by the clicked dimension.
Session expiryA clearly token-related 401 (outside auth endpoints), after an 800 ms grace window with no intervening success, clears the session and returns to /login.

28Exception Handling

ScenarioHandling
Integration auth/permission failureTyped errors (MetaAuthError, MetaPermissionError…) let callers stop or skip a client; UI shows a graceful "billing unavailable" notice; sync continues for other clients.
Rate limiting (Meta/Google)Header parsing detects ≥80% usage; sequential sync, staggered cadence, exponential backoff (2s/8s/32s); overlap guard skips a tick if the prior run is active.
Transient network errorRetried with backoff; cursor pagination capped (≤10,000/edge).
Webhook processing failureEnqueued to MetaWebhookRetry with backoff; poller safety-net; abandoned after final tier.
Process crash mid-syncZombie "running" sync-runs (untouched 30+ min) auto-marked failed at boot before the scheduler restarts.
Concurrent ledger writesIdempotency-key duplicate is caught; the cached balance change rolls back so the net effect is one entry.
Payment recording failureFull rollback of balance + ledger on error (idempotency key payment:<id>).
AI service down / key missingGraceful 503; feature degrades; truncated-JSON responses are repaired before use.
Encryption key missingPortal password login still works (bcrypt); recoverable-copy feature disabled with a logged warning.
Unhandled rejection / uncaught exceptionLogged; server closes gracefully / exits so the orchestrator restarts a clean process.

Maps toCriterion
